# buffa — Architecture Research

Source: `vendors/buffa/` (submodule of `anthropics/buffa`).

Buffa is a pure-Rust protobuf implementation, editions-first, with descriptors as the single input format and a strong codegen pipeline. This document captures the parts that matter when designing `buffa-reflect`.

---

## 2. Schema & wire model

Buffa **never parses `.proto` source itself.** All schema input arrives as `google.protobuf.FileDescriptorProto` produced by `protoc` (≥ v27, with v30+ for editions 2024) or `buf`. The generator operates on already-resolved descriptors with `FeatureSet` resolved per-file/per-field. See `DESIGN.md` and `buffa-codegen/src/lib.rs:1-22`.

Wire format is fully protobuf-compatible: varints, fixed-width, length-delimited, groups, packed/unpacked repeated, maps (synthetic map-entry messages), oneofs, extensions, unknown-field preservation. Conformance suite is the arbiter of correctness.

## 3. `buffa-descriptor` (the bootstrap)

`buffa-descriptor/src/lib.rs` is small — it just re-exports `pub mod generated { pub mod descriptor { ... } pub mod compiler { ... } }`. The interesting bits are the generated types under `buffa-descriptor/src/generated/`:

- `google.protobuf.descriptor.rs` (~426 KB) — `FileDescriptorSet`, `FileDescriptorProto`, `DescriptorProto`, `FieldDescriptorProto` (with `Type` and `Label` enums), `EnumDescriptorProto`, `EnumValueDescriptorProto`, `OneofDescriptorProto`, `ServiceDescriptorProto`, `MethodDescriptorProto`, `FeatureSet`, `SourceCodeInfo`, `*Options`.
- `google.protobuf.compiler.plugin.rs` — `CodeGeneratorRequest`/`CodeGeneratorResponse` for the protoc plugin protocol.

These are full buffa types: `Clone + PartialEq + Default + Message + MessageView`. So we can:

```rust
use buffa::Message;
use buffa_descriptor::generated::descriptor::FileDescriptorSet;
let fds = FileDescriptorSet::decode_from_slice(&bytes)?;
for file in &fds.file {
    for msg in &file.message_type { /* ... */ }
}
```

This is exactly the substrate `buffa-reflect` will need at runtime. **Note**: the buffa runtime trait surface is `Message` / `MessageView` — there is no `prost::Message` on these types.

## 4. Codegen pipeline

### Public entry

`buffa-codegen/src/lib.rs:335` — `pub fn generate(file_descriptors, files_to_generate, config) -> Result<Vec<GeneratedFile>, CodeGenError>`.

Each `.proto` produces five **content files** (`<stem>.rs`, `__view.rs`, `__oneof.rs`, `__view_oneof.rs`, `__ext.rs`); each proto package produces one **stitcher** `<dotted.pkg>.mod.rs`. Build integrations only need to wire up the stitchers; everything else is `include!`-ed transitively. With `CodeGenConfig::file_per_package = true`, the stitcher inlines the content files (used by the BSR SDK packager).

### `CodeGenConfig` shape (`buffa-codegen/src/lib.rs:128-279`)

Relevant knobs we will lean on:

- `extern_paths: Vec<(String, String)>` — proto-path-prefix → Rust-path mapping.
- `bytes_fields: Vec<String>` — promote selected `bytes` fields to `bytes::Bytes`.
- `type_attributes: Vec<(String, String)>` — proto-path-prefix → raw attribute string. **This is the injection point** that lets us add `#[derive(ReflectMessage)]` and `#[buffa_reflect(...)]` to every generated message struct.
- `field_attributes`, `message_attributes`, `enum_attributes` — finer-grained variants.
- `file_per_package: bool`, `emit_register_fn: bool`, `generate_views`, `generate_json`, `generate_text`, `generate_arbitrary`, `preserve_unknown_fields`, `strict_utf8_mapping`, `allow_message_set`.

The `type_attribute` API is verified by `buffa-build/src/lib.rs:308-313` (`Config::type_attribute` forwards to `CodeGenConfig::type_attributes`). Path matching (`buffa-build/src/lib.rs:271-285`) is segment-aware: leading `.` auto-prepended; `"."` matches every type.

### `buffa-build` flow (`buffa-build/src/lib.rs:541-639`)

`Config::compile(self) -> Result<(), Box<dyn Error>>`:

1. Resolve `out_dir` (env `OUT_DIR` if not set).
2. Produce FDS bytes from the configured `DescriptorSource`:
   - `Protoc` (default) — invokes `protoc --descriptor_set_out=...`.
   - `Buf` — invokes `buf build --as-file-descriptor-set`.
   - `Precompiled(PathBuf)` — reads the file as-is. Selected via `Config::descriptor_set(path)` (`buffa-build/src/lib.rs:468`).
3. Decode `FileDescriptorSet` (line 562).
4. Compute `files_to_generate` (proto-relative names; for `Precompiled`/`Buf` modes these are taken verbatim from `.files()`).
5. Call `buffa_codegen::generate(&fds.file, &files_to_generate, &codegen_config)`.
6. Write each `GeneratedFile`. Optionally emit an `_include.rs` module-tree.
7. Print `cargo:rerun-if-changed=…` for the proto inputs.

`DescriptorSource` is a private enum, but its **selectors are public** (`use_buf`, `descriptor_set(path)`), so an outside crate can drive buffa-build via the precompiled-descriptor selector after running protoc/buf itself.

`Config` is consume-by-move: each builder method returns `Self`, and `compile(self)` is terminal. There is no equivalent of prost-build's `skip_protoc_run`. We get the same effect by **doing the protoc invocation ourselves**, then handing the FDS path to `Config::descriptor_set(path).compile()`.

## 5. Generated code shape (existing hooks)

For a typical message (see `vendors/buffa/examples/logging/src/gen/log.v1.log.rs`):

```rust
#[derive(Clone, PartialEq, Default)]
pub struct LogEntry {
    pub timestamp: ::buffa::MessageField<::buffa_types::google::protobuf::Timestamp>,
    pub severity: ::buffa::EnumValue<Severity>,
    pub message: ::buffa::alloc::string::String,
    /* ... */
    #[doc(hidden)]
    pub __buffa_unknown_fields: ::buffa::UnknownFields,
}

impl LogEntry {
    pub const TYPE_URL: &'static str =
        "type.googleapis.com/buffa.examples.log.v1.LogEntry";
}

impl ::buffa::Message for LogEntry { /* compute_size / write_to / merge_field / clear */ }
impl ::buffa::ExtensionSet for LogEntry {
    const PROTO_FQN: &'static str = "buffa.examples.log.v1.LogEntry";
    /* unknown_fields(_mut) */
}
```

Per-message hooks already emitted by codegen:

| Hook | Where | Use |
| --- | --- | --- |
| `const TYPE_URL: &str` | `impl <Msg>` | `Any` packing / type URL lookup. |
| `impl ExtensionSet { const PROTO_FQN }` | message impl | Fully-qualified proto name available at runtime, statically. |
| `merge_field` field-number `match` | `impl Message` | Compile-time wire dispatch, **not** runtime-introspectable. |
| Per-package `__buffa::register_types(&mut TypeRegistry)` | stitcher | Aggregates `Any` and extension entries for JSON/text formats. Gated by `CodeGenConfig::emit_register_fn` (default true). |
| `Enumeration` trait per enum (`buffa/src/enumeration.rs`) | every enum | `from_i32`, `to_i32`, `proto_name`, `from_proto_name`, `values()`. |
| `Extension<C>` const descriptors per extend block | `__buffa::ext::*` | Typed compile-time extension handles. |

What is **not** emitted today:

- No `FileDescriptorSet` artifact in `OUT_DIR`. Codegen consumes descriptors but does not preserve them for the consumer crate.
- No runtime `MessageDescriptor` / `FieldDescriptor` per generated type.
- No reflection trait. `Message::merge_field` is the only field-dispatch path, and its tag table is hardcoded inside the impl.
- No `DynamicMessage`. Explicitly listed as future work (`vendors/buffa/README.md`: "Runtime reflection (DynamicMessage, descriptor-driven introspection) — planned for a future release. The descriptor types are now available in `buffa-descriptor` as a first step.").

## 6. Public API of root `buffa` crate

`use buffa::*` exposes (from `buffa/src/lib.rs`): `Message`, `DefaultInstance`, `MessageField`, `MessageView`, `OwnedView`, `MessageFieldView`, `MapView`, `RepeatedView`, `UnknownFields`, `UnknownFieldsView`, `SizeCache`, `EnumValue`, `Enumeration`, `Extension`, `ExtensionCodec`, `ExtensionSet`, `Oneof`, `DecodeError`, `EncodeError`. Plus the `include_proto!` / `include_proto_relative!` macros.

Feature-gated: `json`, `text`, and `type_registry` (under `json`/`text`).

There is no `Descriptor*` / `ReflectMessage` / `DynamicMessage` in the runtime surface today.

## 7. Project conventions worth honoring

From `vendors/buffa/CLAUDE.md`, `CONTRIBUTING.md`, `DESIGN.md`:

- Pure Rust, zero C deps; descriptor-centric (codegen never parses `.proto`).
- `thiserror` for libraries; no `unwrap` outside tests/safety-comment-bearing `unsafe`.
- Editions-resolved features only; no "is proto2 vs proto3" branching in codegen output.
- Generated code must compile in `no_std` (`extern crate alloc`); type paths emitted as `::buffa::alloc::string::String`, `::buffa::alloc::vec::Vec`, etc.
- API stability: `Message` trait and generated code shape are public; breaking changes require a minor-version bump pre-1.0.
- `CONTRIBUTING.md`: keep PRs ≤ 250 net lines (excluding tests) when possible.

These are buffa's house rules — they apply to anything we upstream into buffa, but `buffa-reflect` will live in its own crate and need only stay compatible.

## 8. Open buffa-side considerations

- **Stability of `buffa-descriptor`**: the crate is generated by buffa-codegen and checked in. Depending on it from `buffa-reflect` is fine (it's a published crate concept), but we must accept that its types track upstream `descriptor.proto` exactly.
- **View types and reflection**: each generated message has a parallel `MessageView<'a>`. A future `DynamicMessage` would need to decide whether it has a view variant; for the build-script crate this question doesn't bind yet.
- **Extension reflection**: extensions today are typed compile-time `Extension<C>` consts. Runtime extension introspection (discovering registered extensions and their value types) needs a per-crate registry, which the codegen-emitted `register_types` already partially solves for JSON/text.
- **Oneof reflection**: oneofs lower to Rust enums under `__buffa::oneof::<msg>`; codegen doesn't expose a runtime "which oneof is set" helper at the message level. Reflection will need this.
