# buffa-reflect Phase 2 — PRD

Phase 1 ([prd](./buffa-reflect-prd.md), [design](./buffa-reflect-design.md), [impl-plan](./buffa-reflect-impl-plan.md)) shipped a build-script + a navigable descriptor pool. With that in place, Phase 2 closes the remaining gaps to **prost-reflect parity** and unlocks the use-cases the PRD originally promised: generic transcoding, schema-aware logging, gRPC server reflection, and SDK-style ergonomics.

This document is the umbrella for Phase 2. Each sub-phase has its own design doc:

| Sub-phase | Design | Status |
| --- | --- | --- |
| 2a — `DynamicMessage` (encode/decode by descriptor, get/set, recursion safety) | [buffa-reflect-dynamic-design.md](./buffa-reflect-dynamic-design.md) | Draft |
| 2b — proto3 canonical JSON via serde (`SerializeOptions`, `DeserializeOptions`, WKT special-cases) | [buffa-reflect-json-design.md](./buffa-reflect-json-design.md) | Draft |
| 2c — textproto encode/decode | [buffa-reflect-text-design.md](./buffa-reflect-text-design.md) | Draft |
| 2d — `grpc.reflection.v1` server shim | [buffa-reflect-grpc-design.md](./buffa-reflect-grpc-design.md) | Draft |
| 2e — view-type reflection (`ReflectMessageView<'a>`) | [buffa-reflect-views-design.md](./buffa-reflect-views-design.md) | Draft |

The body of this PRD pins down the **goals, dependencies, and sequencing** that govern all five.

---

## Problem

The Phase 1 surface gives a downstream consumer everything they need to *describe* a generated message: walk fields by name or number, read enum values, look up a `MessageDescriptor` from a serialized `FileDescriptorSet`. What it does **not** give them:

1. A way to **construct or read a message whose Rust type is not statically known** — required for proto-aware proxies, generic CLIs, schema migration tools, and the bulk of the prost-reflect user base.
2. A canonical **JSON** mapping that interoperates with other protobuf implementations (proto3 JSON spec).
3. A canonical **textproto** mapping (the human-readable format generated by `protoc --decode_raw`, used in many Google internal tools).
4. A **gRPC server reflection** endpoint (`grpc.reflection.v1`), which lets clients discover service signatures at runtime — a hard requirement for any production gRPC service that wants to be discoverable from `grpcurl` / Postman / the buf CLI.
5. **View-type reflection** for the borrowed `*View<'a>` types, so reflection works on the zero-copy decode path too.

Without these, Phase 1 is an incomplete answer to "I want prost-reflect for buffa." Phase 2 closes that gap.

## Users

The same audiences as Phase 1, plus:

- **Service-mesh / observability vendors** who need proto3 JSON to stream messages into log aggregators (Datadog, Honeycomb, Splunk).
- **Operators** who use `grpcurl`/`buf curl` to introspect running services. They cannot work without `grpc.reflection.v1`.
- **CLI tool authors** building generic `bufdump`-style utilities — these need `DynamicMessage` to render arbitrary protos.
- **Migration tooling**: schema diffs, traffic replay across schema versions, all of which need to read messages whose schema is *only* known at runtime.

## Goals

The bar for Phase 2 done:

1. **Phase 2a (`DynamicMessage` core)** — ship a `DynamicMessage` that:
   - decodes any wire-format message given a `MessageDescriptor`,
   - exposes `get_field` / `set_field` / `has_field` / `clear_field` by both proto name and field number,
   - encodes back to wire format with byte-equivalence to the original (when no fields were mutated),
   - preserves unknown fields across decode/encode,
   - integrates with `ReflectMessage` via two new methods: `transcode_to_dynamic(&self) -> DynamicMessage` and `from_dynamic(&DynamicMessage) -> Result<Self, _>`.
2. **Phase 2b (proto3 JSON)** — `serde::Serialize` / `serde::Deserialize` impls for `DynamicMessage` that follow the proto3 JSON mapping byte-for-byte (string-encoded `int64`, base64-encoded `bytes`, RFC3339 `Timestamp`, etc.). `SerializeOptions` / `DeserializeOptions` mirror prost-reflect's surface.
3. **Phase 2c (textproto)** — `DynamicMessage::to_text()` / `DynamicMessage::from_text()` that round-trip through the canonical textproto format.
4. **Phase 2d (gRPC reflection)** — a separate crate `buffa-grpc-reflection` (NOT a member of this workspace by default; a sibling repo or under `examples/`) that implements the `grpc.reflection.v1` service backed by a `DescriptorPool`. Drop-in-compatible with [`tonic-reflection`](https://docs.rs/tonic-reflection)'s setup pattern.
5. **Phase 2e (view-type reflection)** — every generated `*View<'a>` type implements a new `ReflectMessageView<'a>` trait whose `descriptor()` returns the same descriptor as the owned form. Required so observability code on the zero-copy decode path doesn't have to allocate to introspect.
6. **Conformance** — pass the [protobuf conformance suite](https://github.com/protocolbuffers/protobuf/tree/main/conformance) for binary, JSON, and text format roundtrips, in a new crate `buffa-reflect-conformance-tests` mirroring `vendors/prost-reflect/prost-reflect-conformance-tests/`. Failures match the documented "known failures" list and are pinned per release.
7. **Documentation** — every public item carries `///`; the `buffa-reflect` crate's `lib.rs` carries a `# Examples` block with end-to-end transcode (struct → DynamicMessage → JSON → struct).

## Non-goals (Phase 2)

- **`Any` registry beyond the basics.** We provide enough for `Any.unpack()` to work on a known set of WKTs (Timestamp, Duration, etc.) and on user types whose descriptors are in the pool. We do **not** ship a runtime user-type registration mechanism beyond `DescriptorPool::add_file_descriptor_set`.
- **`google.protobuf.Type` / `TypeRegistry` for cross-language type interop.** That's a deeper rabbit hole; `prost-reflect` doesn't ship it either.
- **Custom-options reflection.** Standard descriptors carry option values, but interpreted via runtime reflection only inasmuch as `descriptor_proto()` exposes the raw `*Options` proto. No high-level `extension.get(message)` API.
- **Schema diffing utilities.** Worth building on top, but lives in a different crate.
- **`no_std` for any Phase 2 code.** Same as Phase 1; reconsider once the dust settles.
- **Editions feature inference for packed encoding / repeated_field_encoding.** Phase 1 was conservative ("if features don't tell us, assume proto2 semantics"). Phase 2 inherits that. A future, separate spec covers full editions FeatureSet evaluation.
- **Re-implementation of buffa's encode/decode primitives.** Everywhere we can, we delegate to `buffa::encoding::*`. Phase 2 is descriptor-driven dispatch over those primitives, not a parallel codec.

## Success criteria

- A consumer crate can do this:
  ```rust
  use buffa_reflect::{DescriptorPool, DynamicMessage, ReflectMessage};

  // Read an arbitrary message from the wire.
  let dyn_msg = DynamicMessage::decode(descriptor, &wire_bytes)?;
  println!("name = {:?}", dyn_msg.get_field_by_name("name"));

  // Mutate.
  dyn_msg.set_field_by_name("name", "Alice".into())?;

  // Round-trip back to wire.
  let bytes = dyn_msg.encode_to_vec();

  // JSON.
  let json = serde_json::to_string(&dyn_msg)?;
  let dyn2: DynamicMessage = serde_json::from_str_with_pool(&json, descriptor)?;

  // Transcode between the static type and DynamicMessage.
  let typed: User = ReflectMessage::from_dynamic(&dyn_msg)?;
  let dyn3 = typed.transcode_to_dynamic();
  assert_eq!(dyn3.encode_to_vec(), wire_bytes);
  ```
- For any message in any pool, `dyn_msg.encode_to_vec() == buffa_codec_encode(typed_form)` where the dynamic was produced from the same proto.
- Conformance: ≥ 95% of the protobuf conformance suite passes, with a `known_failures.txt` capturing the known limitations (matching prost-reflect's record).
- Build matrix: `cargo build --workspace --all-features` and `cargo test --workspace --all-features` clean. `cargo clippy --workspace --all-targets -- -D warnings` clean.

## Sub-phase ordering & critical path

```
                     ┌────────────┐
                     │  Phase 2a  │  DynamicMessage core
                     │  (depends  │  (foundational; gates 2b and 2c)
                     │  on Phase 1)│
                     └─────┬──────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         ┌────────┐  ┌──────────┐  ┌──────────┐
         │ 2b JSON │  │ 2c text  │  │  Conf-   │
         │ (serde) │  │  proto    │  │  formance│
         └────────┘  └──────────┘  └──────────┘

Independent (can ship before, after, or in parallel with 2a):

         ┌─────────────────┐    ┌─────────────────┐
         │ 2d gRPC refl.   │    │ 2e view refl.   │
         │ (Phase 1 only)   │    │ (Phase 1 only)  │
         └─────────────────┘    └─────────────────┘
```

The critical path is **2a → (2b ‖ 2c) → conformance**. 2d and 2e have no Phase 2a dependency and can be picked up by parallel contributors or shipped earlier as standalone deliverables.

## Risks

| Risk | Mitigation |
| --- | --- |
| Dynamic encode produces non-canonical wire bytes (different field ordering than the typed encoder), breaking byte-equivalence tests. | Mirror buffa's field-emission order verbatim — iterate fields in `proto.field` declaration order, not alphabetical / number order. Add round-trip tests for every fixture. |
| `DescriptorPool` mutation racing with active `DynamicMessage` references. | `DynamicMessage` clones the pool's `Arc` on construction; mutations on one pool clone are copy-on-write thanks to `Arc::make_mut`. Document explicitly. |
| JSON Any encoding requires `TypeRegistry` to resolve the embedded type. | Provide a default `TypeRegistry` populated from the same `DescriptorPool` the `DynamicMessage` came from, with a hook to override. |
| WKTs (Timestamp, Duration, FieldMask, …) have non-trivial JSON mappings. | Lift the mapping logic from `buffa::json_helpers` where possible (already maintained by upstream); patch only what's missing. |
| Editions semantics drift between `protoc` versions. | Pin `PROTOC_VERSION` in CI; conformance tests gated on the pinned version. Don't try to be more clever than the descriptor — when `FeatureSet` is ambiguous, behave as if proto2. |
| Performance regressions on the typed (Phase 1) hot path from new code in `buffa-reflect`. | Phase 2a goes behind a `dynamic` cargo feature (default on). Consumers who don't want it `default-features = false`. Phase 1 functionality remains in the always-on slice. |
| The `grpc.reflection.v1` shim depends on `tonic` and a transport. | Live in a separate crate with no workspace tie-in. Same pattern as `tonic-reflection`. |

## Out of scope, captured for future work

1. **`DynamicMessage::set_extension` / `Extension` runtime descriptor support** beyond what falls out for free. proto2 extensions reflected through `descriptor_proto()` are readable; first-class get/set is its own spec.
2. **Streaming JSON encode/decode** (`io::Read`/`io::Write` based) — `serde_json::to_writer` / `from_reader` work, but a true streaming-without-allocation path (à la `simd-json`) is its own project.
3. **`MessageView`-flavored `DynamicMessage`** — `DynamicMessageView<'a>` that borrows from a wire buffer. Phase 2a is owned-only; views can be a Phase 3 add-on.
4. **`prost-reflect`-compatible `ReflectMessage::transcode_to_dynamic` performance.** prost-reflect uses unsafe specialization to avoid the encode/decode round-trip in some cases; we explicitly stay safe-Rust at the cost of a single round-trip.

## Migration notes for downstream consumers

- The `dynamic` feature is added with `default-features = ["derive", "dynamic"]`. Existing downstream consumers continue to work unchanged.
- `ReflectMessage` gains two new methods. They are provided via default impls (round-trip through `encode_to_vec`/`decode`), so downstream impls written by hand continue to compile.
- `buffa-reflect-build` learns no new options for Phase 2a-c — the build artifact is unchanged. Phase 2e adds a `generate_view_reflection` knob (default off until view types stabilize on the trait).
